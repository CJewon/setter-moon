# Deployment Guide

SellerRoom의 초기 배포는 Vercel Hobby/free 배포를 기준으로 준비합니다. 실제 유료 서비스로 운영하기 전에는 사용량, 보안, 과금 기준을 다시 확인합니다.

## Vercel 프로젝트 설정

권장 설정:

```txt
Framework Preset: Next.js
Root Directory: front
Install Command: npm install
Build Command: npm run build
Output Directory: 기본값 사용
Node.js Version: 20.9 이상
```

## Vercel 환경 변수

```txt
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<publishable-key>
SUPABASE_SERVICE_ROLE_KEY=<server-only-secret-key>
NEXT_PUBLIC_APP_NAME=SellerRoom
NEXT_PUBLIC_APP_URL=https://<vercel-production-domain>
NEXT_PUBLIC_SUPABASE_STORAGE_BUCKET=product-images
```

주의:

- SUPABASE_SERVICE_ROLE_KEY는 서버 전용 secret입니다. Vercel 프로젝트 환경변수와 GitHub Actions secret에만 등록합니다.
- 브라우저 코드, README, Notion 공개 문서에 secret 값을 쓰지 않습니다.
- E2E_SELLER_EMAIL, E2E_SELLER_PASSWORD는 배포 환경 필수값이 아닙니다.
- 배포 후 자동 smoke test를 Vercel URL로 실행할 때만 별도 CI/로컬 환경에 둡니다.

## Supabase Auth 설정

- Supabase Auth의 Site URL은 Vercel production URL로 설정합니다.
- 이메일 인증, 비밀번호 재설정, OAuth를 실제로 사용할 때는 Supabase Redirect URLs에 production URL, preview URL, localhost URL을 추가합니다.
- 배포 전 Supabase Security Advisor warning을 blocker, warning, 후순위로 분류합니다.

## GitHub Actions 수동 배포

수동 배포는 .github/workflows/vercel-deploy.yml을 사용합니다.

## GitHub Actions 수동 배포 운영 기준

운영 배포 기준은 GitHub Actions의 `Vercel Manual Deploy` workflow를 수동으로 실행하는 방식입니다.

```txt
GitHub > Actions > Vercel Manual Deploy > Run workflow
target: preview | production
run_smoke: true | false
```

Vercel 프로젝트의 Git 자동 배포는 `front/vercel.json`에서 비활성화합니다. 따라서 main 브랜치에 push하더라도 Vercel 대시보드가 자동 배포를 시작하지 않는 것이 정상입니다.

직접 Vercel CLI로 실행하는 `vercel deploy --prebuilt --prod` 배포는 장애 대응, 긴급 복구, 원인 분석 같은 emergency/debug 용도로만 사용합니다. 일반 운영 배포 이력은 GitHub Actions 실행 기록을 기준으로 확인합니다.

workflow는 build/deploy 전에 GitHub Actions에 등록된 앱 환경변수를 검사합니다. GitHub Actions 값은 Vercel 프로젝트 환경변수로 동기화한 뒤 사용합니다.

아래 값은 GitHub Actions 기준으로 누락되거나 비어 있으면 실패 처리합니다.

```txt
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
SUPABASE_SERVICE_ROLE_KEY
```

이 단계는 secret 값을 로그에 출력하지 않고 키 이름과 누락/빈 값 여부만 표시합니다. 검증이 끝나면 workflow가 `vercel env add --force`를 stdin 입력으로 실행해 GitHub Actions 값을 Vercel 프로젝트 환경변수에 동기화합니다.

`SUPABASE_SERVICE_ROLE_KEY`는 `--sensitive` 옵션으로 동기화합니다. secret 값은 `.vercel/.env.production.local`, `.vercel/.env.preview.local`, command-line argument에 직접 쓰지 않습니다.

동기화 후 `vercel pull`로 프로젝트 설정을 가져오되, 앱 환경변수 3개는 로컬 `.vercel/.env.*.local` 파일에서 제거합니다. build는 GitHub Actions step `env:`로 앱 환경변수를 주입한 상태에서 `vercel build`를 실행해 파일에 secret을 남기지 않습니다. deploy는 `vercel deploy --prebuilt`만 실행하며 런타임 값은 Vercel 프로젝트 환경변수를 사용합니다.

참고한 공식 문서:

```txt
Vercel GitHub Actions guide: https://vercel.com/guides/how-can-i-use-github-actions-with-vercel/
Vercel CLI deploy: https://vercel.com/docs/cli/deploy
Next.js on Vercel: https://vercel.com/docs/concepts/next.js/overview
GitHub Actions workflows: https://docs.github.com/en/actions/concepts/workflows-and-actions/workflows
```

GitHub repository secrets:

```txt
VERCEL_TOKEN
SUPABASE_SERVICE_ROLE_KEY
E2E_SELLER_EMAIL
E2E_SELLER_PASSWORD
VERCEL_AUTOMATION_BYPASS_SECRET
```

GitHub repository variables:

```txt
VERCEL_ORG_ID
VERCEL_PROJECT_ID
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
```

주의:

- Vercel 프로젝트의 Environment Variables와 GitHub repository secrets는 서로 다른 설정입니다.
- GitHub Actions에서 Vercel CLI를 실행하려면 GitHub repository secrets에 VERCEL_TOKEN이 반드시 있어야 합니다.
- GitHub Actions에서 `vercel build`와 `vercel deploy --prebuilt`를 안정적으로 실행하려면 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`를 GitHub Actions에도 등록해야 합니다. workflow가 이 값을 Vercel 프로젝트 환경변수로 동기화합니다.
- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`는 브라우저 공개값이므로 GitHub repository variables로 등록하는 것을 기본으로 합니다. secrets에 등록해도 workflow는 동작합니다.
- `SUPABASE_SERVICE_ROLE_KEY`는 서버 전용 secret이므로 반드시 GitHub repository secret으로 등록합니다.
- Vercel Deployment Protection이 켜진 URL에서 smoke test를 실행하려면 VERCEL_AUTOMATION_BYPASS_SECRET도 GitHub repository secret에 등록합니다. 보호 설정이 없으면 선택값입니다.
- VERCEL_ORG_ID와 VERCEL_PROJECT_ID는 GitHub repository variables에 등록하는 것을 기본으로 합니다. secrets에 등록해도 workflow는 동작합니다.
- Actions 로그에 `--token=`처럼 토큰 값이 빈 상태로 보이면 VERCEL_TOKEN이 등록되지 않았거나 이름이 다르게 등록된 상태입니다.
- VERCEL_TOKEN은 Vercel Account Settings > Tokens에서 발급한 값을 GitHub repository secret으로 등록합니다.
- VERCEL_ORG_ID와 VERCEL_PROJECT_ID는 Vercel 프로젝트 연결 정보에서 확인한 값을 GitHub repository variable로 등록합니다.
- GitHub 등록 위치는 GitHub repository > Settings > Secrets and variables > Actions 입니다.
- 로컬에서 `npx.cmd vercel@latest link --cwd front`를 실행하면 현재 Vercel CLI 기준으로 `.vercel/repo.json`이 생성됩니다.
- `.vercel/repo.json`의 `projects[0].orgId`를 VERCEL_ORG_ID로, `projects[0].id`를 VERCEL_PROJECT_ID로 등록합니다.

실행 방법:

```txt
GitHub > Actions > Vercel Manual Deploy > Run workflow
```

입력값:

```txt
target: preview | production
run_smoke: 배포 후 deploy smoke E2E 실행 여부
```

운영 기준:

- 현재는 main 브랜치만 배포합니다.
- preview는 기능 확인, production은 실제 배포에 사용합니다.
- 공식 Vercel GitHub Actions 가이드와 맞춰 Build and Deploy job 하나에서 Vercel build와 Vercel deploy --prebuilt를 연속 실행합니다.
- 한 job에서 실행하는 이유는 Vercel prebuilt 산출물이 .vercel/output, front/.next, node_modules 같은 로컬 빌드 산출물을 함께 참조할 수 있기 때문입니다.
- Build와 Deploy를 별도 job으로 나누면 artifact 전달 과정에서 hidden file, Next 빌드 산출물, 런타임 의존성이 누락될 수 있어 현재 프로젝트에서는 사용하지 않습니다.
- Build and Deploy job에서 typecheck, lint, test, Vercel build, Vercel deploy --prebuilt를 순서대로 실행합니다.
- Deploy 단계는 Vercel CLI 공식 옵션인 --archive=tgz를 사용해 prebuilt 산출물을 압축 업로드합니다. 런타임 앱 환경변수는 deploy command line이 아니라 Vercel 프로젝트 환경변수에서 읽습니다.
- run_smoke를 켜면 Deploy 이후 Smoke Test job이 추가로 실행됩니다.
- front/vercel.json에서 Vercel Git 자동 배포를 꺼두었으므로, push만으로는 배포되지 않습니다.
- 나중에 develop, stage, main 브랜치를 분리하면 workflow 입력값에 배포 브랜치를 다시 추가합니다.
