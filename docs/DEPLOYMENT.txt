# Deployment Guide

SellerRoom의 초기 배포는 Vercel Hobby/free 배포를 기준으로 준비합니다. 실제 유료 서비스로 운영하기 전에는 사용량, 보안, 과금 기준을 다시 확인합니다.

## Vercel 프로젝트 설정

권장 설정:

```txt
Framework Preset: Next.js
Root Directory: front
Install Command: npm install
Build Command: npm run build
Output Directory: 기본값 사용
Node.js Version: 20.9 이상
```

## Vercel 환경 변수

```txt
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=<publishable-key>
SUPABASE_SERVICE_ROLE_KEY=<server-only-secret-key>
NEXT_PUBLIC_APP_NAME=SellerRoom
NEXT_PUBLIC_APP_URL=https://<vercel-production-domain>
NEXT_PUBLIC_SUPABASE_STORAGE_BUCKET=product-images
```

주의:

- SUPABASE_SERVICE_ROLE_KEY는 Vercel 환경 변수에만 등록합니다.
- 브라우저 코드, README, Notion 공개 문서에 secret 값을 쓰지 않습니다.
- E2E_SELLER_EMAIL, E2E_SELLER_PASSWORD는 배포 환경 필수값이 아닙니다.
- 배포 후 자동 smoke test를 Vercel URL로 실행할 때만 별도 CI/로컬 환경에 둡니다.

## Supabase Auth 설정

- Supabase Auth의 Site URL은 Vercel production URL로 설정합니다.
- 이메일 인증, 비밀번호 재설정, OAuth를 실제로 사용할 때는 Supabase Redirect URLs에 production URL, preview URL, localhost URL을 추가합니다.
- 배포 전 Supabase Security Advisor warning을 blocker, warning, 후순위로 분류합니다.

## GitHub Actions 수동 배포

수동 배포는 .github/workflows/vercel-deploy.yml을 사용합니다.

GitHub repository secrets:

```txt
VERCEL_TOKEN
E2E_SELLER_EMAIL
E2E_SELLER_PASSWORD
```

GitHub repository variables:

```txt
VERCEL_ORG_ID
VERCEL_PROJECT_ID
```

주의:

- Vercel 프로젝트의 Environment Variables와 GitHub repository secrets는 서로 다른 설정입니다.
- GitHub Actions에서 Vercel CLI를 실행하려면 GitHub repository secrets에 VERCEL_TOKEN이 반드시 있어야 합니다.
- VERCEL_ORG_ID와 VERCEL_PROJECT_ID는 GitHub repository variables에 등록하는 것을 기본으로 합니다. secrets에 등록해도 workflow는 동작합니다.
- Actions 로그에 `--token=`처럼 토큰 값이 빈 상태로 보이면 VERCEL_TOKEN이 등록되지 않았거나 이름이 다르게 등록된 상태입니다.
- VERCEL_TOKEN은 Vercel Account Settings > Tokens에서 발급한 값을 GitHub repository secret으로 등록합니다.
- VERCEL_ORG_ID와 VERCEL_PROJECT_ID는 Vercel 프로젝트 연결 정보에서 확인한 값을 GitHub repository variable로 등록합니다.
- GitHub 등록 위치는 GitHub repository > Settings > Secrets and variables > Actions 입니다.
- 로컬에서 `npx.cmd vercel@latest link --cwd front`를 실행하면 현재 Vercel CLI 기준으로 `.vercel/repo.json`이 생성됩니다.
- `.vercel/repo.json`의 `projects[0].orgId`를 VERCEL_ORG_ID로, `projects[0].id`를 VERCEL_PROJECT_ID로 등록합니다.

실행 방법:

```txt
GitHub > Actions > Vercel Manual Deploy > Run workflow
```

입력값:

```txt
target: preview | production
run_smoke: 배포 후 deploy smoke E2E 실행 여부
```

운영 기준:

- 현재는 main 브랜치만 배포합니다.
- preview는 기능 확인, production은 실제 배포에 사용합니다.
- GitHub Actions 화면에서는 Build -> Deploy 순서로 분리되어 보입니다.
- Build job에서 typecheck, lint, test, Vercel build를 실행합니다.
- Build job은 .vercel/output과 .vercel/project.json을 1일 보관 artifact로 넘깁니다.
- .vercel/output/functions 안의 .vc-config.json 파일은 점으로 시작하는 hidden file이므로 upload-artifact에서 include-hidden-files를 켜야 합니다.
- Deploy job은 repository를 checkout하고 npm ci로 의존성을 설치한 뒤 build artifact를 내려받아 vercel deploy --prebuilt 방식으로 배포합니다.
- Deploy job에서도 checkout을 수행하는 이유는 Vercel 프로젝트의 Root Directory가 `front`라서, prebuilt 배포 중에도 runner 안에 `front` 경로가 존재해야 하기 때문입니다.
- Deploy job에서도 npm ci를 수행하는 이유는 Vercel CLI가 prebuilt 함수 파일을 업로드/검증하면서 node_modules의 런타임 의존성을 참조할 수 있기 때문입니다.
- run_smoke를 켜면 Deploy 이후 Smoke Test job이 추가로 실행됩니다.
- front/vercel.json에서 Vercel Git 자동 배포를 꺼두었으므로, push만으로는 배포되지 않습니다.
- 나중에 develop, stage, main 브랜치를 분리하면 workflow 입력값에 배포 브랜치를 다시 추가합니다.
